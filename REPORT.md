### 1. Ссылка на реализацию требования: "При добавлении поста у друга, лента должна обновляться автоматически (с небольшой задержкой)"

Реализация выполнена через паттерн **Deferred Materialization (Отложенная материализация)**. Задержка достигается за счет асинхронной обработки очереди.

*   **Логика:**
    1.  Сервис слушает очередь `post_processing_queue`.
    2.  При получении сообщения имитирует задержку обработки (если раскомментировать `asyncio.sleep`).
    3.  Находит список друзей автора.
    4.  Отправляет событие в RabbitMQ с ключом маршрутизации (Routing Key), равным ID друга.

[ссылка на код](services/processor-service/main.py#L31)
```python
async for message in queue_iter:
    async with message.process():
        data = json.loads(message.body)
        author_id = data.get("user_id")
        # ... поиск друзей ...
        for friend_id in friends:
            # Отправка события конкретному другу
            await feed_exchange.publish(msg, routing_key=str(friend_id))
```

### 2. Ссылка на реализацию требования: "Корректная работа сервиса вебсокетов"

Корректная работа обеспечивается управлением жизненным циклом соединения: подключение, обработка сообщений, корректное отключение и очистка ресурсов.

*   **Логика:**
    1.  При подключении пользователь "биндится" (подписывается) на события в RabbitMQ.
    2.  Сервис слушает входящие сообщения от RabbitMQ и пробрасывает их в WebSocket.
    3.  При разрыве соединения (`WebSocketDisconnect`) происходит "анбинд" (отписка), чтобы не засорять очередь сообщениями для отключенного пользователя.

[ссылка на код](services/ws-service/main.py#L79)
```python
@app.websocket("/ws/feed/{user_id}")
async def websocket_endpoint(websocket: WebSocket, user_id: int):
    await ws_manager.connect(websocket, user_id)
    
    # Подписка на события конкретного пользователя
    await rabbit_listener.bind_user(user_id)

    try:
        while True:
            # Keep-alive цикл
            data = await websocket.receive_text()
            # ...
    except WebSocketDisconnect:
        ws_manager.disconnect(user_id)
        # Важный момент: очистка подписки в RabbitMQ
        await rabbit_listener.unbind_user(user_id)
```

### 3. Описание процесса масштабируемости сервиса вебсокетов

Сервис вебсокетов спроектирован **Stateless** (без сохранения состояния на уровне приложения), что позволяет масштабировать его линейно путем добавления новых инстансов.

**Механизм масштабирования (Dynamic Routing):**

1.  **Отсутствие общего состояния:** Каждый инстанс сервиса (`ws-service`) при запуске создает свою уникальную (эксклюзивную) очередь в RabbitMQ. Список подключенных пользователей хранится только в оперативной памяти конкретного инстанса.
2.  **Динамическая подписка (Binding):**
    *   Когда пользователь `User_A` подключается к `Server_1`, `Server_1` отправляет в RabbitMQ команду: *"Привяжи мою очередь к Exchange с ключом `User_A`"*.
    *   RabbitMQ теперь знает, что все сообщения с ключом `User_A` нужно отправлять в очередь `Server_1`.
3.  **Горизонтальное масштабирование:**
    *   Если запустить `Server_2`, он создаст свою очередь.
    *   Если `User_A` переподключится и попадет на `Server_2`, он создаст привязку там.
    *   RabbitMQ автоматически маршрутизирует сообщение на тот сервер, где сейчас находится пользователь.
    *   **Результат:** Можно запустить 10, 50, 100 инстансов сервиса. Load Balancer (Nginx) будет распределять новые подключения равномерно, а RabbitMQ будет доставлять сообщения точно в нужный инстанс.

### 4. Описание процесса масштабирования RabbitMQ

RabbitMQ является центральным узлом обмена сообщениями. Для поддержки высокой нагрузки применяется **Кластеризация**.

**Этапы масштабирования:**

1.  **Кластеризация (Clustering):**
    *   Запускается несколько узлов RabbitMQ, которые объединяются в единый кластер.
    *   Все узлы знают друг о друге. Exchange и очереди видимы глобально.

2.  **Балансировка очередей (Queue Master Location):**
    *   По умолчанию мастер-очередь создается на узле, где её объявили.
    *   При масштабировании используется политика `x-queue-master-locator: client-local` или `balanced`. Это заставляет очереди, создаваемые разными WebSocket-серверами, распределяться по разным физическим узлам RabbitMQ, выравнивая нагрузку на сеть и CPU.

3.  **Quorum Queues (Надежность):**
    *   Используются очереди типа `quorum` вместо классических `classic`. Они гарантируют сохранность данных (Durability) при падении узлов, так как данные реплицируются между узлами с помощью алгоритма Raft.

4.  **Load Balancer:**
    *   Перед кластером стоит TCP-балансировщик (HAProxy).
    *   FastAPI сервисы (API, Processor, WS) подключаются к адресу балансировщика, а не к конкретным узлам. Это позволяет прозрачно добавлять новые узлы RabbitMQ или выводить их из эксплуатации для обслуживания.
